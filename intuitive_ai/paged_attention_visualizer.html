🚀 PagedAttention: Virtual Memory for KV Cache

The Big Idea: Apply operating system virtual memory techniques to KV cache management.

Just like your OS uses paging to efficiently manage RAM, vLLM uses PagedAttention to efficiently manage GPU memory for KV cache.

Result: 10-20x better throughput with the same hardware!

⚠️ Part 1: The Traditional Approach (And Its Problems)

Let's first see how traditional inference systems (like HuggingFace Transformers) manage KV cache memory.

Problem 1: Static Pre-allocation

Traditional systems allocate memory based on the maximum sequence length, not the actual length needed.

Example Scenario:

  • Model supports max 4096 tokens
  • User request: 100-token prompt, generates 50 tokens = 150 tokens total
  • Memory allocated: 4096 tokens worth (27x more than needed!)
  • Memory actually used: 150 tokens worth
  • Wasted: 96% of allocated memory
Efficiency = actual_tokens / max_tokens = 150 / 4096 = 3.6%

Problem 2: Memory Fragmentation

Even with dynamic allocation (allocating exactly what's needed), memory becomes fragmented over time.

Fragmentation Example:

Step 1: Allocate contiguous memory for 3 requests

Request B (800 tokens)
Request C (600 tokens)

Free

Step 2: Request B finishes → leaves a 800-token hole

Free

Step 3: New Request D needs 1000 tokens

WASTED (800)

Free (100)
❌ Problem: Request D needs 1000 tokens, but:
  • The 800-token hole isn't big enough
  • The 100-token free space isn't big enough
  • Total free = 900 tokens, but they're fragmented
  • Can't allocate contiguous 1000 tokens!

800 tokens of perfectly usable memory sits wasted!

The Real-World Impact

Metric Traditional Approach Impact
Memory Efficiency 20-40% 60-80% of GPU memory wasted
Batch Size 8-16 requests Can't serve many users simultaneously
Throughput Baseline Low requests/second
Cost Efficiency High $/token Expensive to operate

💡 Part 2: Inspiration from Operating Systems

Before diving into PagedAttention, let's understand how operating systems solve this exact problem for RAM.

How Your OS Manages Memory

Operating systems use virtual memory with paging to efficiently manage RAM:

Key Concepts:

1. Fixed-Size Pages

Memory is divided into fixed-size blocks called pages (typically 4KB).

Page 0
Page 1
Page 2
Page 3
...

2. Virtual Addresses → Physical Addresses

Programs use virtual addresses (contiguous), OS maps them to physical addresses (can be non-contiguous).

Process thinks: "My data is at addresses 0-7KB (contiguous)"
Reality: Page 0 → Physical frame 5, Page 1 → Physical frame 2 (non-contiguous!)

3. Page Table

A lookup table that maps virtual pages to physical frames.

Virtual Page 0 → Physical Frame 5
Virtual Page 1 → Physical Frame 2
Virtual Page 2 → Physical Frame 9
Virtual Page 3 → Physical Frame 1

4. On-Demand Allocation

Pages are allocated only when needed, not upfront.

🎯 Why This Eliminates Fragmentation

  • Fixed-size blocks: Any free page can be used for any request
  • Non-contiguous physical storage: A process's "contiguous" virtual memory can use scattered physical pages
  • No wasted space: Internal fragmentation is limited to the last page only
  • Flexible allocation: Allocate/deallocate pages dynamically

Key insight: Physical memory doesn't need to be contiguous if we have a mapping layer!

Visual Example: OS Paging

Virtual Address Space (Process View)

Contiguous and simple

Virtual Page 0 (0-4KB)
Virtual Page 1 (4-8KB)
Virtual Page 2 (8-12KB)

The process sees contiguous memory, but physical frames are scattered → No fragmentation!

🚀 Part 3: PagedAttention - Applying Paging to KV Cache

vLLM applies the exact same paging concept to KV cache management. Brilliant!

The Core Idea

Instead of storing KV cache for each request in contiguous GPU memory, divide the KV cache into fixed-size blocks (like OS pages) and store them in non-contiguous locations, using a block table (like page table) to track them.

Key Components:

OS Paging PagedAttention Description
Page Block Fixed-size unit (e.g., 16 tokens)
Virtual Address Logical Block ID Sequential identifier (Block 0, 1, 2...)
Physical Frame Physical Block Actual GPU memory location
Page Table Block Table Maps logical blocks to physical blocks
Process Request/Sequence Each has its own mapping

Block Structure

Each block stores KV cache for a fixed number of tokens across all layers.

Block (16 tokens) = {
  For each layer (e.g., 40 layers) {
    For each token (0-15) {
      Key vectors (all attention heads)
      Value vectors (all attention heads)
    }
  }
}

Block Size Example (Llama 2 13B, 16 tokens/block)

Block_size = tokens_per_block × cache_per_token
= 16 × 819,200 bytes
= 13,107,200 bytes
≈ 12.5 MB per block

Common block sizes: 8, 16, or 32 tokens per block

🔍 Part 5: How Attention Works with Paged Blocks

The critical question: How does the model perform attention when K,V are scattered across non-contiguous blocks?

The Challenge

Traditional attention assumes KV cache is stored contiguously:

# Traditional attention (pseudo-code) def traditional_attention(query, kv_cache): # kv_cache is a single contiguous tensor # shape: [num_tokens, num_layers, num_heads, head_dim] keys = kv_cache[..., 0, :] # Extract all keys values = kv_cache[..., 1, :] # Extract all values # Compute attention scores scores = query @ keys.T weights = softmax(scores) output = weights @ values return output
❌ Problem: With PagedAttention, keys and values are scattered across non-contiguous blocks! We need a different approach.

PagedAttention's Solution

Use the block table to gather K,V from non-contiguous blocks during attention computation.

# PagedAttention (simplified pseudo-code) def paged_attention(query, block_table, physical_blocks): # block_table: maps logical blocks to physical blocks # physical_blocks: actual GPU memory containing KV cache num_blocks = len(block_table) outputs = [] # Process each block for logical_block_id in range(num_blocks): # Look up physical block location physical_block_id = block_table[logical_block_id] # Fetch K,V from this physical block block_kv = physical_blocks[physical_block_id] keys = block_kv[..., 0, :] values = block_kv[..., 1, :] # Compute attention for this block scores = query @ keys.T weights = softmax(scores) block_output = weights @ values outputs.append(block_output) # Combine outputs from all blocks final_output = combine(outputs) return final_output

🎯 Key Insights:

  • Indirect access: Use block_table[logical_id] → physical_id
  • Block-wise computation: Process attention one block at a time
  • Same result: Final output is mathematically identical to traditional approach
  • GPU kernel optimization: vLLM implements this efficiently in CUDA

Performance Considerations

⚡ Part 6: Advanced PagedAttention Features

1. Block Sharing for Parallel Sampling

When generating multiple completions for the same prompt (beam search, parallel sampling), blocks can be shared!

Scenario: Generate 3 different completions for the same prompt

Prompt: "Once upon a time" (10 tokens = 1 block)

Traditional Approach:

Store 3 copies of the prompt KV cache

Completion 1
Prompt Block (copy 1)
Completion 2
Prompt Block (copy 2)
Completion 3
Prompt Block (copy 3)

Total: 3 blocks for prompt + completion blocks = wasteful!

PagedAttention with Block Sharing:

All 3 completions point to the SAME physical prompt block

Completion 1
Gen Block 1
↓
Completion 2
Gen Block 2
↓
Completion 3
Gen Block 3
↓
Shared Prompt Block (stored once!)

Total: 1 prompt block + 3 generation blocks = 2x memory savings!

How It Works:

  • Prompt blocks are marked read-only with reference counting
  • All completions' block tables point to the same physical prompt blocks
  • Only the generated tokens need unique blocks per completion
  • When all completions finish, reference count → 0, block freed

2. Copy-on-Write for Beam Search

When a beam branches, share blocks until they diverge, then copy only when needed.

Beam Search Example:

2 beams share the same sequence until token 20, then diverge.

Tokens 0-16: Both beams identical

Both beam block tables point to the same physical blocks (shared, read-only)

Token 17-20: Still identical

Continue sharing blocks

Token 21: Beams diverge!

Copy-on-write: Allocate new block only for the diverging beam, keep sharing previous blocks

Benefit: Only copy blocks when beams actually diverge, not upfront. Massive memory savings for beam search!

3. Dynamic Block Allocation

Blocks are allocated on-demand as tokens are generated, not pre-allocated.

Approach When Memory Allocated Amount Allocated Waste
Traditional At request start Max possible (e.g., 4096 tokens) 60-80%
PagedAttention As tokens generated One block at a time (e.g., 16 tokens) ~5-10%*

* Only the last block may be partially empty

✅ Key Takeaways: Why PagedAttention Dominates

The Core Innovations

1. Fixed-Size Blocks

Divide KV cache into fixed-size blocks (e.g., 16 tokens). Any free block can serve any request → no fragmentation.

2. Block Table (Virtual Memory)

Map logical blocks (contiguous from request's view) to physical blocks (scattered in GPU memory). Enables non-contiguous storage.

3. On-Demand Allocation

Allocate blocks only as tokens are generated, not upfront. No wasted pre-allocation.

4. Block Sharing

Multiple sequences can share read-only blocks (e.g., prompt blocks in parallel sampling). Massive memory savings.
