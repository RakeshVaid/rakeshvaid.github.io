📚 Part 1: Attention Mechanism Refresher

Before diving into KV cache, let's quickly review how attention works in transformers.

Query (Q)

What I'm looking for

Generated from the current token being processed. It asks: "What information do I need from previous tokens?"

Key (K)

What I contain

Generated from each token. It represents: "Here's what information I have to offer."

Value (V)

My actual content

Generated from each token. It's the actual information to be retrieved if the Key matches the Query.

🔑 The Attention Formula

Attention(Q, K, V) = softmax(Q × KT / √dk) × V

In plain English:

  1. Compare the Query with all Keys (Q × KT) to get attention scores
  2. Normalize with softmax to get attention weights
  3. Use those weights to mix the Values

🎬 Step-by-Step Matrix Walkthrough

Let's see exactly how attention works when generating the next token. We'll use a simple 3-token example with small dimensions for clarity.

Scenario: Generate the next word after "The cat"

Input tokens: ["The", "cat"]
Goal: Generate the next token (predict "sat")

Step 1: Token Embeddings → Q, K, V Projections

Each token gets transformed into Query, Key, and Value vectors through learned weight matrices.

For simplicity, let's say each vector has dimension 4 (in reality, it's 128+ per attention head)

Token Query (Q) Key (K) Value (V)
"The" [0.2, 0.5, 0.1, 0.8] [0.3, 0.7, 0.2, 0.4] [0.9, 0.1, 0.5, 0.3]
"cat" [0.6, 0.3, 0.9, 0.2] [0.5, 0.4, 0.8, 0.1] [0.2, 0.7, 0.4, 0.6]

Step 2: Compute Attention for "cat"

To generate the next token, we need to compute attention from "cat" to all previous tokens (including itself).

Qcat = [0.6, 0.3, 0.9, 0.2]
KT = [[0.3, 0.5], ← "The" and "cat" keys
         [0.7, 0.4],
         [0.2, 0.8],
         [0.4, 0.1]]

Compute attention scores (Q × KT):

ScoreThe = Qcat · KThe = 0.6×0.3 + 0.3×0.7 + 0.9×0.2 + 0.2×0.4 = 0.77
Scorecat = Qcat · Kcat = 0.6×0.5 + 0.3×0.4 + 0.9×0.8 + 0.2×0.1 = 1.14

Apply softmax to get attention weights:

Attention_weights = softmax([0.77, 1.14]) = [0.34, 0.66]

This means "cat" pays 34% attention to "The" and 66% to itself

Weighted sum of Values:

Output = 0.34 × VThe + 0.66 × Vcat
Output = 0.34 × [0.9, 0.1, 0.5, 0.3] + 0.66 × [0.2, 0.7, 0.4, 0.6]
Output = [0.438, 0.496, 0.434, 0.498]

Step 3: Generate Next Token "sat"

Now we need to generate "sat". The model creates Q, K, V for this new position.

Token Query (Q) Key (K) Value (V)
"The" — [0.3, 0.7, 0.2, 0.4] [0.9, 0.1, 0.5, 0.3]
"cat" — [0.5, 0.4, 0.8, 0.1] [0.2, 0.7, 0.4, 0.6]
"sat" (new) [0.4, 0.8, 0.3, 0.5] [0.6, 0.2, 0.7, 0.9] [0.3, 0.5, 0.8, 0.2]

Compute attention from "sat" to all previous tokens:

Qsat = [0.4, 0.8, 0.3, 0.5]

ScoreThe = Qsat · KThe = 0.4×0.3 + 0.8×0.7 + 0.3×0.2 + 0.5×0.4 = 0.94
Scorecat = Qsat · Kcat = 0.4×0.5 + 0.8×0.4 + 0.3×0.8 + 0.5×0.1 = 0.81
Scoresat = Qsat · Ksat = 0.4×0.6 + 0.8×0.2 + 0.3×0.7 + 0.5×0.9 = 1.16

Attention_weights = softmax([0.94, 0.81, 1.16]) = [0.31, 0.27, 0.42]

💡 The Critical Insight

Notice what we need at each generation step:

  • To generate "cat": We need K and V for "The"
  • To generate "sat": We need K and V for "The" AND "cat"
  • To generate next token: We need K and V for "The", "cat", AND "sat"

❌ Without Caching:

At each step, we'd recompute K and V for ALL previous tokens:

Step 1: Compute K,V for "The"
Step 2: Compute K,V for "The" + K,V for "cat" ← recomputing "The"!
Step 3: Compute K,V for "The" + K,V for "cat" + K,V for "sat" ← recomputing everything!

For a 100-token generation, we'd compute "The" 100 times!

✅ With KV Cache:

Store K and V once, reuse them forever:

Step 1: Compute K,V for "The" → CACHE IT
Step 2: Load cached K,V for "The", compute K,V for "cat" → CACHE IT
Step 3: Load cached K,V for "The" + "cat", compute K,V for "sat" → CACHE IT

Result: Each token's K,V is computed exactly once → 50-100x speedup!

🧮 What Gets Cached Per Token?

For each token at each layer:

Cached_data = {
  Key_vector: [k₁, k₂, k₃, ..., kd], ← dimension d (e.g., 128 per head)
  Value_vector: [v₁, v₂, v₃, ..., vd] ← dimension d
}

Why not cache Query? Because Q is only used for the current token's attention computation, then discarded. Only K and V need to persist for future tokens to attend to.

📊 Real-World Example: Llama 2 13B Cache Structure

Let's calculate exactly how many KV vectors are stored for a real model architecture.

Llama 2 13B Architecture:

Parameter Value Symbol
Number of Layers 40 N
Attention Heads per Layer 40 m
Dimension per Head 128 d
Total Hidden Size 5120 (= 40 heads × 128) h
Precision FP16 (2 bytes) p

For a Single Token:

At each layer, each token has:

• m attention heads = 40 heads
• Each head has: K vector (dim d) + V vector (dim d)
• Total vectors per layer = 40 heads × 2 (K,V) = 80 vectors
• Each vector dimension = 128 floats

Memory per token per layer:

= m heads × 2 (K,V) × d dimension × p bytes
= 40 × 2 × 128 × 2 bytes
= 20,480 bytes
= 20 KB per layer

Memory per token across ALL layers:

= N layers × 20 KB
= 40 × 20 KB
= 800 KB
= ~0.8 MB per token

For a Complete Sequence:

General Formula:

KV_cache_size = 2 × N × m × d × p × num_tokens

Or equivalently:
KV_cache_size = 2 × N × h × p × num_tokens

Where h = m × d (total hidden size)

Llama 2 13B Examples:

Sequence Length Calculation Total KV Cache
1 token 2 × 40 × 5120 × 2 × 1 ~0.8 MB
512 tokens 2 × 40 × 5120 × 2 × 512 ~400 MB
2048 tokens (2K) 2 × 40 × 5120 × 2 × 2048 ~1.6 GB
4096 tokens (4K) 2 × 40 × 5120 × 2 × 4096 ~3.2 GB
8192 tokens (8K) 2 × 40 × 5120 × 2 × 8192 ~6.4 GB

🔑 Key Insights:

  • Linear scaling: KV cache grows linearly with sequence length
  • Per request: Each concurrent user needs their own cache
  • All layers: Every transformer layer maintains its own K,V for every token
  • Both K and V: We store 2 vectors (Key + Value) per head
  • All heads: Multi-head attention means m×d total dimensions to cache

Example bottleneck:

80GB A100 GPU with Llama 2 13B:

• Model weights: 26 GB
• Available for KV cache: 54 GB
• Max 2K-token requests: 54 GB ÷ 1.6 GB = ~33 concurrent requests
• Max 4K-token requests: 54 GB ÷ 3.2 GB = ~16 concurrent requests

This is why KV cache management is critical for serving!

Comparison Across Model Sizes:

Model Layers (N) Heads (m) Dim (d) Hidden (h) Cache/Token Cache/2K tokens
Llama 2 7B 32 32 128 4096 ~0.5 MB ~1.0 GB
Llama 2 13B 40 40 128 5120 ~0.8 MB ~1.6 GB
Llama 2 70B 80 64 128 8192 ~2.5 MB ~5.0 GB
Mixtral 8x7B 32 32 128 4096 ~0.5 MB ~1.0 GB

Note: MoE models like Mixtral have the same KV cache size as their base counterpart since KV cache depends on hidden size, not total parameters.

🎯 Part 2: Why Do We Need KV Cache?

LLMs generate text autoregressively - one token at a time. Each new token must attend to ALL previous tokens.

🔍 Without KV Cache (Naive Approach)

Imagine generating: "The cat sat"

Step 1: Generate "cat"
Process: "The" → Compute K,V for "The" → Generate "cat"
Step 2: Generate "sat"
Process: "The cat" → Compute K,V for "The" again, compute K,V for "cat" → Generate "sat"
❌ Problem: We recompute K and V for "The" at every step! This is incredibly wasteful.

✅ With KV Cache (Smart Approach)

Instead, we cache the K and V vectors for all previous tokens.

Step 1: Generate "cat"
Process: "The" → Compute and CACHE K,V for "The" → Generate "cat"
Step 2: Generate "sat"
Process: "cat" only → Use CACHED K,V for "The", compute and cache K,V for "cat" → Generate "sat"
✅ Benefit: We only compute K and V once per token! Massive speedup (50-100x faster).

🧮 Part 4: The Memory Math

Let's break down exactly how much memory the KV cache consumes.

Memory Formula per Token

Memory_per_token = 2 × num_layers × hidden_size × precision_bytes

Where:

  • 2 = K and V vectors (we need both)
  • num_layers = Number of transformer layers (e.g., 32, 40, 80)
  • hidden_size = Dimension of the hidden state (e.g., 4096, 5120, 8192)
  • precision_bytes = 2 for FP16, 4 for FP32

Example: Llama 2 13B

Memory_per_token = 2 × 40 layers × 5120 × 2 bytes = 819,200 bytes ≈ 800 KB per token

For a Full Sequence

Total_KV_cache = Memory_per_token × sequence_length
Sequence Length KV Cache Size (13B model)
512 tokens ~400 MB
2048 tokens (2K context) ~1.6 GB
4096 tokens (4K context) ~3.2 GB
32768 tokens (32K context) ~25.6 GB

⚠️ Part 5: Why This Is THE Bottleneck

Model Weights vs KV Cache

Let's compare what takes up GPU memory during inference:

Component Size (13B model) Per Request? Notes
Model Weights ~26 GB (FP16) ❌ No (shared) Loaded once, shared across all requests
KV Cache ~1.6 GB per 2K tokens ✅ Yes Unique for each request
Activations ~100-200 MB ✅ Yes Small compared to KV cache

🔥 The Key Insight

Model weights are loaded once and shared. They don't scale with the number of requests.

KV cache scales linearly with:

  • Number of concurrent requests
  • Length of each request (prompt + generation)

Result: For batch inference, KV cache dominates memory usage!

Real World Example

80GB A100 GPU running Llama 2 13B:

Available for KV cache = 80 GB - 26 GB (weights) = 54 GB
Max concurrent 2K-token requests = 54 GB / 1.6 GB ≈ 33 requests

Without efficient KV cache management, you can only serve ~30 users simultaneously!

💸 Part 6: How Traditional Systems Waste Memory

Problem 1: Static Pre-allocation

Traditional inference engines (like HuggingFace's default) allocate memory based on the maximum possible sequence length.

Example: If your model supports 4096 max tokens:
❌ Each request reserves 4096 tokens worth of memory (~3.2 GB)
❌ Even if the actual generation is only 100 tokens (needs ~80 MB)
❌ Wasted memory: ~3.1 GB per request!

Problem 2: Memory Fragmentation

Even with dynamic allocation, memory becomes fragmented:

  • Request A: 1000 tokens
  • Request B: 2000 tokens
  • Request A finishes → 1000 token "hole" in memory
  • New Request C needs 1500 tokens → can't fit in the 1000 token hole
  • Result: Wasted memory gaps

The Impact

Metric Traditional Approach Impact
Memory Efficiency 20-40% 60-80% of allocated memory is wasted
Batch Size Small (8-16) Can't serve many concurrent users
Throughput Low Expensive per-token cost

🚀 Part 7: vLLM's Solution - PagedAttention

The Breakthrough Idea

Inspiration from Operating Systems: How does your OS manage RAM efficiently?

  • Virtual memory with paging
  • Divide memory into fixed-size pages
  • Map virtual addresses to physical pages
  • No fragmentation, efficient allocation

🎯 PagedAttention applies the same concept to KV cache!

Instead of allocating contiguous memory for each request:

  1. Divide KV cache into fixed-size blocks (e.g., 16 tokens per block)
  2. Allocate blocks on-demand as tokens are generated
  3. Use a page table to track which blocks belong to which request
  4. Non-contiguous blocks in physical memory → no fragmentation!

Memory Efficiency Comparison

Traditional (HF) ~30% ~15 requests 1x (baseline)
vLLM (PagedAttention) ~90% ~100+ requests 10-20x

✅ Key Takeaways

1. KV Cache is the Bottleneck

During LLM inference, KV cache memory grows linearly with:

  • Number of tokens (prompt + generation)
  • Number of concurrent requests

It quickly dominates memory usage for batch serving.

2. Why We Need KV Cache

Without caching K and V vectors, we'd recompute them for every token generation → 50-100x slower.

3. Traditional Systems Waste Memory

Static allocation and fragmentation mean 60-80% of allocated memory sits unused.

4. vLLM's Innovation

PagedAttention uses virtual memory techniques to achieve ~90% memory efficiency → 10-20x more throughput on the same hardware.
